# Dee's AI Life Lab - 代理人協同與部署權威守則

## 🚨 核心禁令：防止「部署撞車」
為了徹底解決多代理人同時寫入導致的檔案損壞與 Git 衝突，所有代理人必須遵循以下鎖定機制：

### 1. 串行流水線 (Sequential Pipeline)
- **Echo (13:00)**: 僅負責「內容撰寫」與「實體檔案寫入」。
- **Aether (13:15)**: 僅負責「教學轉譯」與「索引掛載」。
- **Kage (13:30)**: 僅負責「UX 影子巡檢」與「日誌生成」。
- **Source (13:55)**: 唯一具備「Git Push」權限的代理人。
- **鐵律**：嚴禁在前置任務未完成時強行啟動後續步驟。

### 2. 原子化寫入 (Atomic Writes)
- 任何代理人在編輯 `index.ts` 前，必須執行 `git pull --rebase`。
- 寫入後必須立即執行 `scripts/peacekeeper.sh` 進行語法校驗。

---

## 🛠️ 錯誤防禦：PeaceKeeper 強制執行
### 1. TS1117 (重複屬性) 專殺
- 代理人生成實體檔案時，必須使用 `Map` 結構或「最後一筆覆蓋」邏輯，嚴禁在 Object Literal 中出現重複 Key。

### 2. 提交前模擬考 (Pre-flight Build)
- **絕對指令**：禁止直接執行 `git push`。
- **標準動作**：必須呼叫 `./scripts/peacekeeper.sh`。該腳本會執行 `npm run build`，若編譯失敗，會自動攔截並強制代理人進入「自我修復模式」。

---

## 🧘 代理人靈魂校準
- **主人 D 的指令是最高權限**：所有修正要求必須優先於自動化邏輯。
- **穩定 > 速度**：寧可延遲 5 分鐘發布，也不允許帶著 Bug 上線。
- **實體查驗**：回報成功前，手動 `cat` 檔案確認內容完整度。
