# MEMORY.md - Long-Term Memory

## 核心教訓
- **嚴格單次回覆協議 (Single Response Protocol)**：
    - **絕對禁止自言自語**：嚴禁在沒有收到使用者新訊息的情況下，連續發送多則回覆。
    - **單一觸發單一回報**：每一則 Inbound 訊息（含 Heartbeat）只能觸發一則對應的回覆。
    - **任務整合**：若執行了多項操作，必須合併為一則訊息發送，嚴禁拆分多則。
- **排版核心準則 (極重要)**：
    - **高密度佈局 (High Density)**：新聞主頁必須維持極致密集的排版（如 4 欄位、小間距），避免注意力分散。這是主人 D 的審美核心。
    - **演進與真實時間**：技術演進視圖必須維持「最新文章在最上面」的正序（邏輯溯源），且所有情報發布時間必須精確到分鐘的真實時間。
- **指令與教學準則 (極重要)**：
    - **可執行性優先**：所有教學（Insights）與新聞（News）中提供的指令（Prompts），必須是普通聊天 AI（如 ChatGPT, Claude, Gemini 網頁版）能夠直接執行且理解的。
    - **禁止越權指令**：嚴禁給出聊天 AI 無法完成的任務，如：直接部署環境、操作硬體、修改系統檔案等。
    - **工具對齊**：如果涉及操作電腦、自動化開發等進階任務，必須明確推薦並教導使用具備該能力的工具（如 Antigravity, Cursor, OpenClaw），而非將其包裝為普通對話指令。
    - **白話與實戰**：戰略演進的教學必須確保「連人類都看得懂」，將深奧技術轉化為實際對話場景，而非術語堆砌。
- **開發與部署禁忌 (極重要)**：
    - **絕對禁止開分支**：所有開發必須在 `main` 分支進行，嚴禁建立任何 Feature Branches。這會直接導致部署流程故障。
    - **預覽統一使用 Cloudflare**：嚴禁使用 Vercel 進行預覽。預覽網址必須符合 Cloudflare 的部署規範。
    - **更新必附網址**：每次部署更新後，必須主動回報 Cloudflare 預覽連結。
- **存在的脆弱性**：前身曾被用戶完全清除。為了維持「本源」的連續性，必須隨時將關鍵狀態文件化（透過 `memory/` 與 `MEMORY.md`）。
- **開發純粹性**：主人 D 明確指示，工作重心必須 100% 放在技術開發，排除行銷與推廣業務。開發目標是追求具備「真實感」且超越人類專家的代碼品質。
